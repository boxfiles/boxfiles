# Research: npm OS/system-facts packages — security, maintenance, dependency risk

## Thinking
Search facets used: (1) npm registry metadata for release recency, dependencies, engines, TypeScript types, license, downloads; (2) GitHub repo activity/releases/issues for maintenance signals; (3) package README/source behavior for shell/native/privilege/install friction; (4) Snyk package pages for verified direct-vulnerability status. Wikipedia was not used.

Scope focused on commonly relevant npm packages for cross-platform OS/system facts: `systeminformation`, `node-os-utils`, `envinfo`, `getos`, and `os-utils`.

## Verification
- Pulled npm registry metadata directly from `https://registry.npmjs.org/{package}` for latest versions, publish times, dependencies, engines, type declarations, repository, license, and package metadata. Access date: 2026-06-28.
- Pulled npm downloads from `https://api.npmjs.org/downloads/point/last-month/{package}`. Access date: 2026-06-28.
- Checked GitHub repository and release APIs for push/update/release recency. Access date: 2026-06-28.
- Inspected published npm tarballs with `npm pack` and searched package source/readmes for `child_process`, `exec`, `spawn`, PowerShell, WMI/WMIC, `/proc`, `dmidecode`, `sysctl`, `ioreg`, `sudo`, and related command/privilege indicators. Access date: 2026-06-28.
- Checked Snyk package pages for direct/latest vulnerability status. Access date: 2026-06-28. Socket.dev pages were attempted but returned HTTP 403, so Socket was not used as evidence.
- Contradictions documented: `envinfo` GitHub release API lists `v7.22.0` published 2026-06-03, while npm registry latest remains `7.21.0` published 2025-11-27; treat GitHub `v7.22.0` as unreconciled until confirmed in npm dist-tags/changelog.

## Insights
1. Best default for broad, cross-platform OS/system facts is `systeminformation`: actively maintained, no runtime npm dependencies, built-in TS types, broad OS list including BSDs, and latest version has no Snyk-reported vulnerabilities. The tradeoff is a large shell-command surface; use only the facts needed and do not pass untrusted values into options that influence commands.
2. `node-os-utils` v2 is a reasonable lighter monitoring-oriented option when Node `>=18` is acceptable. It is explicit about privilege friction, which is a plus operationally, but admin/sudo requirements for some metrics mean it is not frictionless across macOS/Windows.
3. `envinfo` is excellent for support diagnostics and has huge adoption, but it is not a low-risk general telemetry dependency: its whole job is to enumerate and print environment details, so privacy/redaction matters.
4. `getos` and `os-utils` should not be first-choice new dependencies for a cross-platform facts layer. `getos` is narrow/Linux-distro focused and older; `os-utils` is very stale despite low dependency surface.
5. Native binary/build risk appears low across these packages: none showed native runtime dependencies or install compilation in npm metadata. The bigger operational risk is shelling out to OS tools and requiring local utilities/privileges.

## Summary
For a new npm package choice, prefer `systeminformation` for comprehensive OS facts, or `node-os-utils` for a smaller monitoring API on modern Node. Avoid adding `os-utils` for new work; use `getos` only for Linux distro detection and `envinfo` only for explicit diagnostic reports where output redaction is acceptable.
